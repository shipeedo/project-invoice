# Office 365 / Microsoft Graph

Mailbox intake uses a Microsoft Entra (Azure AD) app registration to access a shared Office 365 mailbox via Microsoft Graph.

## Environment variables

| Variable | Required | Notes |
|----------|----------|-------|
| `MS_CLIENT_ID` | Yes (Phase 1b) | Application (client) ID from Azure app registration |
| `MS_CLIENT_SECRET` | Yes (Phase 1b) | Client secret — **never commit** |
| `MS_TENANT_ID` | Yes (Phase 1b) | Directory (tenant) ID |

Add these to **`.env`** locally. Use your host's secret store in production.

## What is not in env

Per the PRD, the **shared mailbox address** (e.g. `invoices@yourdomain.com`) is configured in the **application UI**, not as an environment variable. This allows changing the monitored mailbox without redeploying.

## Authentication flow

The app uses **client credentials** (app-only) or delegated access depending on implementation — typical for a background mailbox poller:

1. Authenticate with `MS_CLIENT_ID`, `MS_CLIENT_SECRET`, `MS_TENANT_ID`
2. Obtain a Microsoft Graph access token for the tenant
3. Read new messages from the configured shared mailbox
4. On new email → trigger AI agent to process attachments

Token endpoint:

```
https://login.microsoftonline.com/{MS_TENANT_ID}/oauth2/v2.0/token
```

## Required Graph permissions (expected)

Exact permissions depend on mailbox type; commonly:

| Permission | Type | Purpose |
|------------|------|---------|
| `Mail.Read` | Application | Read mail in shared mailbox (app-only) |
| or `Mail.Read.Shared` | Application | Read shared mailboxes |

Confirm granted admin consent on the app registration for the chosen permission set.

## Implementation notes

- New email arrival triggers the invoice processing agent
- Raw email and attachments must be stored and visible in the portal
- Mailbox connection settings should be validated from the admin UI
