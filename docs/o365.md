# Office 365 / Microsoft Graph

Office 365 mailbox intake is **configured per tenancy in the UI** — not via environment variables. An admin connects their Microsoft 365 organization and selects which shared mailbox to monitor.

## Who does what

| Actor | Capability |
|-------|------------|
| **Admin role** | Connect/disconnect Office 365, select or change the shared mailbox |
| **All tenancy users** | Access invoices sourced from the connected shared mailbox |

Once a shared mailbox is connected, **every user in the tenancy** can view and work with invoices received via that mailbox. No per-user mailbox setup.

## Connect flow (UI)

```
Admin (any user with admin role) → Settings → Connect Office 365
        ↓
Microsoft OAuth consent (admin/delegated — TBD at implementation)
        ↓
Connection stored for this tenancy (tokens + tenant ID in DB)
        ↓
Fetch available shared mailboxes via Microsoft Graph
        ↓
Admin selects mailbox → saved as tenancy mailbox config
        ↓
Background poller/webhook monitors selected mailbox
        ↓
New email → AI agent processes attachments → visible to all tenancy users
```

## Stored per tenancy (database — not env)

| Field | Purpose |
|-------|---------|
| Microsoft tenant ID | Customer's Entra directory |
| Access / refresh tokens | Graph API access (encrypted at rest) |
| Selected mailbox ID / UPN | e.g. `invoices@customer.com` |
| Connection status | Connected, disconnected, error |
| Connected by / connected at | Audit |

## Platform env vars (Shipeedo app registration only)

The **project-invoice platform** needs a multi-tenant Azure app registration to power the "Connect Office 365" button. These are **platform-level** secrets (one set for the whole product), not per-customer config:

| Variable | Notes |
|----------|-------|
| `MS_CLIENT_ID` | Azure app (client) ID for the Connect flow |
| `MS_CLIENT_SECRET` | Azure client secret — **never commit** |

Do **not** put customer `MS_TENANT_ID` or mailbox credentials in env — those come from the UI connect flow and are stored per tenancy.

Optional platform config:

| Variable | Notes |
|----------|-------|
| `MS_REDIRECT_URI` | OAuth callback for Microsoft connect, e.g. `http://localhost:3000/api/auth/callback/microsoft` |

Microsoft authorize URL uses the multi-tenant endpoint:

```
https://login.microsoftonline.com/organizations/oauth2/v2.0/authorize
```

Token exchange uses the tenant returned from the connect flow, not a fixed env tenant ID.

## Required Graph permissions (expected)

Permissions depend on connect model (delegated vs application). Likely candidates:

| Permission | Type | Purpose |
|------------|------|---------|
| `Mail.Read` | Delegated or Application | Read mail in the selected shared mailbox |
| `Mail.Read.Shared` | Delegated or Application | Access shared mailboxes |
| `User.Read` | Delegated | Identify connecting user during OAuth |
| `offline_access` | Delegated | Refresh tokens for background polling |

Admin consent may be required when an admin connects Office 365.
